How it Works

One protocol, one memory store. Here’s the simple flow.

1

Install & Init

Run npm install -g @memlink/cli then memlink init to create a memory with a unique 12-character ID.

2

Serve via MCP

memlink serve starts a Streamable HTTP server at http://localhost:4444/mcp — the Model Context Protocol endpoint your agent talks to.

3

Connect Any Agent

Drop the MCP URL into your agent’s config (Claude, Cursor, OpenCode, etc.). Now every agent shares the same persistent memory.

CLI at a Glance

Every command is designed to be simple, predictable, and self-documenting.

Create a Memory

Kickstart your persistent memory layer with a single command. memlink init generates a unique 12-character memory ID, initializes the underlying storage file, and prints the ready-to-use MCP URL. The optional --serve flag even auto-starts the server so you can connect immediately — zero to connected in under a second.

Start the Server

memlink serve launches the MCP server using Express and the official @modelcontextprotocol/sdk, broadcasting every registered memory on the Streamable HTTP transport. Use --cors * to allow cross-origin requests from web UIs, or --read-only to enforce a strict no-write policy — ideal for production deployments where data integrity is critical.

Connection Details

Need the MCP URL for a specific memory? memlink connect looks up the memory by its 12-character ID, prints the full name, ID, and Streamable HTTP endpoint, and automatically copies the URL to your clipboard — ready to paste straight into your agent’s opencode.json or claude_desktop_config.json.

WSL Bridge

Windows users running OpenCode or Devin inside WSL face a classic networking hurdle: WSL2’s localhost is isolated from Windows. memlink wsl-connect solves this by running ip route show default inside WSL to automatically detect the host gateway IP, then constructs a reachable MCP URL you can drop straight into your WSL agent’s config — no manual IP hunting required.
